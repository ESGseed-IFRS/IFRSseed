
**배포 구조 (Vercel + EC2 + Git + Docker Hub + Neon pgvector)**

- **Vercel**: `vercel.json` 없음. `www.ifrsseed.com`은 Next.js이고 README에 “Deploy on Vercel”만 있음 → 프론트는 Vercel 배포 가능하나, 리포 안에 Vercel 전용 설정은 없음.
- **EC2**: 배포 단위는 `docker-compose.yaml`. EC2용 설정/스크립트는 없고, 인프라 쪽에서 compose만 돌리는 구조로 보임.
- **Git / CI**: `.github/workflows` 등 CI로 Docker Hub 푸시·EC2 배포하는 설정은 검색 결과에 없음 → 이미지 푸시/배포는 수동이거나 다른 저장소/도구에 있을 수 있음.
- **Docker Hub**: 각 서비스에 `Dockerfile` 있음 (gateway, user, oauth, ifrs_agent 등). compose는 `build: context/dockerfile`로 로컬 빌드만 하고, **이미지 이름/레지스트리(Docker Hub) 지정이 없음** → 푸시/태깅은 CI 또는 수동 스크립트에 따로 있을 가능성.
- **Neon DB**  
  - Java 서비스(user, oauth, common): `NEON_DATABASE_URL`, `NEON_DATABASE_USER`, `NEON_DATABASE_PASSWORD` 사용 (`application.yaml` / compose).  
  - Python(ifrs_agent, alembic, load 스크립트): `DATABASE_URL` 사용 (`base.py`, `env.py`, 스크립트들).  
  → 배포 시 **둘 다** 넣어야 함 (Python용 `DATABASE_URL`, Java용 `NEON_*`).
- **pgvector**: `vector_db.py`에서 확장 활성화, `models.py`에서 `pgvector.sqlalchemy.Vector`, alembic 004에서 벡터 컬럼. Neon은 pgvector 지원하므로 구조상 문제 없음.

**UTF-8**

- **파일/스크립트**: `.env` 로드는 `encoding='utf-8'` (실패 시 utf-16), JSON/CSV 등 `open(..., encoding='utf-8')` 다수 사용.
- **표준 입출력**: `ifrs_agent/main.py`에서 `sys.stdout/stderr.reconfigure(encoding='utf-8')` 설정됨.
- **DB**: `create_engine(DATABASE_URL)`, `psycopg2.connect(database_url)`에 **charset/encoding 인자 없음**. PostgreSQL/Neon 기본이 UTF-8이라 보통 문제 없음. 한글 깨짐이 있으면 URL에 `?options=-c%20client_encoding%3Dutf8` 추가하거나 연결 후 `SET client_encoding = 'UTF8'` 고려.

**요약**

- 배포 구조(역할 분리)와 Neon·pgvector 사용은 맞게 되어 있음.  
- Vercel/EC2/Docker Hub를 코드로 쓰는 부분(vercel 설정, CI로 이미지 푸시·배포)은 리포에 없거나 다른 위치에 있을 수 있음.  
- UTF-8은 파일/콘솔 쪽으로 쓰고 있고, DB는 기본 UTF-8이라 별도 지정은 없음.







지금 코드/구성만 보면 **치명적인 문제는 없고**, 나중에 한 번씩만 점검하면 될 것들만 정리해 두었습니다.

---

**나중에 확인해 두면 좋은 것들**

1. **환경 변수 통일**  
   배포(EC2 등)에서 Python 서비스가 쓰는 DB URL을 **하나로** 쓸지 정해 두기.  
   - Neon 쓰면: `DATABASE_URL`에 Neon connection string 넣고,  
   - Java는 JDBC용으로 `NEON_DATABASE_URL` 등 그대로 두면 됨.  
   두 값이 같은 DB를 가리키는지 한 번만 확인.

2. **Docker 이미지 푸시**  
   Docker Hub에 올리는 흐름이 있으면,  
   - 어떤 이미지 이름/태그로 푸시하는지,  
   - EC2에서 그 이미지를 pull해서 쓰는지  
   (또는 EC2에서 `docker-compose build`만 하는지) 한 번만 정리해 두기.

3. **DB 인코딩**  
   운영 DB에서 한글이 깨지면,  
   - Neon 대시보드에서 DB/프로젝트 기본 인코딩이 UTF-8인지 확인하고,  
   - 필요하면 연결 URL에 `?options=-c%20client_encoding%3Dutf8` 붙이기.

4. **data_integration 서비스 (SR 에이전트)**  
   지금은 compose에 없고, EC2에서 따로 띄우는 구조라면  
   - 포트(예: 9005),  
   - `GROQ_API_KEY`, `TAVILY_API_KEY` 등 필요한 env  
   가 배포 스크립트/문서에 들어가 있는지 한 번만 확인.

지금 단계에서는 “문제 없다”고 보면 되고, 위 네 가지는 배포/운영 단계에서 체크 리스트로 쓰면 됩니다.
