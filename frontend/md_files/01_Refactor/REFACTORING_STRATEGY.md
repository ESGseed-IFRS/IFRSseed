# 프론트엔드 리팩토링·구조 가이드

작성일: 2026-03-26  
목적: **파일당 1000줄 미만**, **폴더·파일명만으로 역할 파악**, **신규 수정·구현 시 동일 규칙 준수**. 문서·코드 모두 **짧고 일관되게** 유지해 사람·도구(에디터/AI) 모두의 부담을 줄인다.

---

## 1. 현재 구조 스냅샷(참고)

```
src/
├── app/                          # Next.js App Router
│   ├── layout.tsx, page.tsx      # 루트 셸·랜딩
│   └── (main)/                   # 메인 앱 라우트 그룹
│       ├── ghg_calc/             # GHG 산정·감사·리포트 도메인
│       │   ├── page.tsx          # 탭 라우팅(얇게 유지)
│       │   ├── types/
│       │   ├── lib/              # 상수·mock·순수 유틸
│       │   └── components/
│       │       ├── layout/       # GHG 전용 레이아웃·사이드바
│       │       ├── ghg/          # 산정 플로우 UI
│       │       ├── audit/        # 감사·결재·라인리지 등 (+ tabs/, data/, types/)
│       │       ├── raw-data/     # 원자료 업로드·동기화
│       │       └── report/       # GHG 보고서 출력
│       └── sr-report/            # SR 보고서 워크스페이스
│           ├── page.tsx
│           ├── lib/
│           └── components/
├── components/                   # 앱 전역(홈·로그인·네비 등) + ui/
├── hooks/
├── lib/                          # 전역 api, utils
└── store/                        # 전역 상태(필요 시만)
```

**원칙**: 기능은 가능하면 **`app/(main)/<기능명>/` 아래에 모은다.** 여러 화면에서 쓰는 것만 `src/components`, `src/lib`, `src/hooks`로 올린다.

---

## 2. 하드 규칙

| 규칙 | 설명 |
|------|------|
| **파일 길이** | 단일 파일 **목표 400줄 이하**, **절대 1000줄 초과 금지**. 초과 시 분리 필수. |
| **이름** | `PascalCase.tsx` = 컴포넌트, `camelCase.ts` = 훅·상태·유틸, `types.ts` / `*.types.ts` = 타입 묶음. |
| **한 파일 한 책임** | 한 파일에 “페이지 조립 + API + 거대한 JSX + mock 데이터”를 넣지 않는다. |
| **데이터·상수** | mock·데모·큰 상수 배열은 `lib/mock*.ts`, `data/*.ts`로 분리. |
| **UI 분할** | 탭·모달·섹션은 하위 컴포넌트 또는 `tabs/` 같은 **하위 폴더**로. |

---

## 3. 직관적인 폴더 규칙

1. **라우트(`page.tsx`)는 얇게**  
   상태(선택된 탭 등)와 자식 컴포넌트 연결만. 실제 화면은 `components/`로.

2. **도메인 단위 콜로케이션**  
   GHG 감사 전용 타입·mock·워크플로는 `ghg_calc/components/audit/` 또는 `ghg_calc/lib/`에 둔다. SR만 쓰는 것은 `sr-report/`에만 둔다.

3. **하위 기능 폴더 명**  
   - `layout/` — 셸·사이드바·헤더  
   - `tabs/` — 탭별 패널만  
   - `data/` — mock·데모 데이터·타임라인 빌더  
   - `types/` — DTO·이벤트 타입 (재사용 시 `ghg_calc/types`로 승격 검토)

4. **`components/ui/`**  
   shadcn 등 **디자인 시스템 primitives**. 비즈니스 로직·도메인 API 호출을 넣지 않는다.

---

## 4. 토큰·유지보수 효율(짧은 코드·적은 맥락)

- **거대 단일 파일 금지**: AI·리뷰어가 전체를 읽어야 하면 토큰·시간이 폭증한다.  
- **깊은 `index.ts` 남발 금지**: `export *` 다단 재export는 import 경로 추적을 어렵게 한다. 필요한 곳에서 **직접 경로 import**를 우선한다.  
- **주석은 “왜”만**: 무엇을 하는지는 이름으로 드러내고, 비즈니스 제약·주의만 한 줄.  
- **타입은 가까이**: 해당 기능 폴더 `types/`에 두고, 2곳 이상에서 쓰일 때만 공통 위치로 이동.  
- **문서(md)도 짧게**: 이 파일처럼 **원칙·체크리스트 중심**. 장문 스펙은 기능별 `md_files/02_...`에만.

---

## 5. 분리 체크리스트(1000줄·400줄 넘기 전)

- [ ] JSX 블록이 **같은 패턴의 섹션**으로 3회 이상 반복 → `*Section.tsx` 또는 `tabs/Tab*.tsx`  
- [ ] `useState`/`useEffect`가 **서로 다른 concern**(fetch + 폼 + 테이블) → 커스텀 훅 `use*.ts` (같은 폴더 또는 `hooks/`)  
- [ ] 상수·라벨·컬럼 정의가 **50줄 이상** → `constants.ts` 또는 `columns.ts`  
- [ ] 순수 함수(정렬·집계·포맷) → `lib/*.ts`  
- [ ] 페이지가 **switch로 큰 분기만** → `CalcContent.tsx`처럼 **콘텐츠 전용 파일**로 추출 (이미 `ghg_calc/page.tsx` 패턴 유지)

---

## 6. 신규 기능·수정 시 워크플로

1. **어느 라우트/도메인인지** 먼저 정한다 → `app/(main)/<도메인>/` 아래에만 추가할지, 공통으로 올릴지 결정.  
2. **새 파일명**이 폴더 동료들과 **같은 패턴**인지 확인 (`Approval*`, `RawData*`, `GHG*` 등).  
3. **한 PR(또는 한 커밋)에서** “파일 분리 + 동작 유지”를 우선하고, 스타일만 손대는 리팩터는 분리.  
4. 분리 후 **import 경로**가 `../../../` 3단 이상이면 `lib/` 또는 `components/` 하위 구조를 한 단계 정리(폴더 이동) 검토.  
5. **전역 store**는 정말 여러 라우트에서 필요할 때만; 나머지는 로컬 state + props 또는 React Query 등 데이터 레이어로.

---

## 7. 금지·지양

- 단일 컴포넌트 파일에 **500줄 이상 JSX** (분할 전제 없이 추가 금지).  
- `utils.ts`에 **도메인별 로직** 누적 → `ghg_calc/lib`, `sr-report/lib` 등으로 분산.  
- UI 컴포넌트에 **raw fetch + 복잡한 reduce** 혼재 → 훅 또는 `lib/api` 쪽으로.  
- 의미 없는 접두사(`New`, `Final`, `V2`) 남용 → 기존 파일 정리 또는 이름을 역할 기반으로.

## 8. 이 문서의 역할

- 팀·본인이 **리팩토링 우선순위**를 잡을 때 기준으로 쓴다.  
- **새 구현** 시 “어디에 파일을 둘지·언제 쪼갤지” 빠르게 합의한다.  
- 필요 시 기능별 상세 설계는 `md_files/02_Dashboard/...` 등에 두고, **이 문서는 교차 참조만** 링크로 연결한다.
