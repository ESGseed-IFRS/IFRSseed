# 지주 SR 작성 대시보드 ↔ 실제 SR 보고서(지주 모드) 연동·확장 전략

> **대상 화면**  
> 1) **국내 자회사/사업장 × DP 제출 현황** — KPI + (가상의) 대형 매트릭스  
> 2) **지주 보고서 페이지 현황** — 실제 `h-write` 페이지 목록과 동일한 축으로 진행률·업무 처리  
>
> **이 문서가 다루는 실무 과제(요약)**  
> - DP(열)가 **수십~수백**으로 늘어도 **한 화면에서 운용 가능한가**  
> - **제출** 상태면 **제출 본문(문단)을 보고** 승인/반려까지 이어지게 할 것인가  
> - **미제출**이면 **독촉(리마인드)** 를 바로 보낼 것인가  
> - **페이지 현황**은 네모 카드에 얽매이지 않고, **실제 지주 SR「페이지별 작성」목록**과 맞출 것인가  

관련: `SR_DASHBOARD_APPROVAL_AND_REPORT_UNIFICATION_STRATEGY.md` (통합 결재함), 계열사 DP 본문은 `/sr-report` 계열사 모드.

---

## 1. 역할 분리(수정)

| 구분 | 지주 SR **대시보드** | **SR 보고서** (`/sr-report` · `mode=holding`) |
|------|----------------------|-----------------------------------------------|
| 목적 | **조직·DP·페이지 단위로 막힘을 드러내고**, 독촉·검토 진입점 제공 | **본문 편집·취합·확정** |
| 데이터 | 집계·스냅샷 + **검토용 요약 본문**(읽기) | 원문·버전·에디터 상태(쓰기) |

**원칙(완화):**  
- “대시보드에 미니 에디터를 두지 않는다”는 **풀 편집**에는 여전히 유효하다.  
- 다만 **제출·검토 업무**를 위해, 셀/행에서 열리는 **읽기 전용 미리보기(문단·첨부 메타)** + **승인/반려/결재함 이동**은 **대시보드·드로어·사이드 패널**에서 허용한다. (결재 **본문 처리**의 단일 허브는 통합 결재함과 정합할 것.)

---

## 2. 공통 전제: 보고서 컨텍스트 키

- `fiscalYear` / `reportCycleId`, `holdingEntityId`, `workspace=holding`  
- 백엔드 전에는 mock·로컬이라도 **키 스키마를 고정**해 두고 API로 치환한다.

---

## 3. 1번 영역: DP가 매우 많아질 때의 정보 설계

### 3.1 왜 “전열 매트릭스”만으로는 한계인가

- 열(DP)이 늘어날수록 **가로 스크롤·가독성·성능**이 동시에 악화된다.  
- “모든 DP를 한 번에 다 보여준다”는 목표 자체를 **운영 화면**에서 버리고, **탐색·필터·요약 + 상세**로 나눈다.

### 3.2 권장 UI 패턴(조합 가능)

1. **기본 뷰: 요약 + 상세 드로어(또는 우측 패널)**  
   - 화면 중앙은 **행(조직) 중심 테이블** 또는 **열(DP) 중심 테이블** 중 하나만 “메인”으로 두고, 나머지 축은 **검색·필터·접기**로 처리한다.  
   - **셀 클릭** 시 전체 매트릭스를 유지하지 않고, **선택 조직·선택 DP**에 대한 **상세 패널**을 연다.

2. **DP 축 처리**  
   - **DP 그룹**(환경/사회/거버넌스, GRI/ISSB/ESRS 등) 탭 또는 트리로 **열 수를 논리적으로 줄인 뷰**.  
   - **자주 보는 DP / 내 담당 DP**만 고정 열로 두고, 나머지는 “더보기” 또는 검색으로 연다.  
   - **가상 스크롤(virtualized)** 로 가로·세로 대형 그리드의 DOM 부담을 줄인다.

3. **피벗 전환**  
   - **조직 우선**: 한 조직을 선택하면 해당 조직의 DP 목록이 **세로 리스트**로 펼쳐진다(열 스크롤 문제 완화).  
   - **DP 우선**: 한 DP를 선택하면 조직별 상태가 **세로 리스트**로 펼쳐진다.

4. **히트맵 + 숫자**  
   - 전체 그리드는 **색상·숫자만** 작게 보여 주고, **텍스트·승인**은 항상 **드로어**에서 한다.

### 3.3 데이터·API 관점

- **목록 API**: 페이지네이션 또는 `entityId`/`dpPackageId` 범위 스캔.  
- **셀 상세 API**: `entityId + dpPackageId` 단위로 **상태 + 제출 요약 + 최신 본문 스니펫 + 결재 문서 id**를 내려준다.  
- 전 그리드를 한 번에 내려주는 방식은 **DP·조직 수가 커지면 비현실적**이므로, 집계는 KPI/히트맵용, 상세는 **온디맨드**로 가져온다.

## 4. 1번 영역: 셀별 업무 — 제출 본문 보기 · 승인/반려 · 미제출 독촉

### 4.1 상태별 행동(실무 시나리오)

| 셀 상태(예시) | 사용자가 하고 싶은 일 | 전략 |
|---------------|----------------------|------|
| **미제출** | 담당자에게 독촉 | **독촉** 버튼 → 채널(메일/알림) + 템플릿 + 이력 저장. 행 단위·일괄 독촉과 연동. |
| **작성 중** | 진행만 확인 / 가볍게 독촉 | 요약·마지막 저장 시각 표시, 필요 시 동일 독촉 플로우. |
| **제출·검토 중** | **제출한 문단(또는 요약)을 보고** 승인/반려 | **읽기 전용 미리보기**에 `latestSubmissionText`(또는 스니펫) + 첨부 링크. **승인/반려**는 (a) 통합 결재함의 해당 문서로 딥링크하거나, (b) 권한·정책상 허용 시 **즉시 승인/반려 API** 후 결재함과 동기화. |
| **반려** | 사유 확인 후 재제출 유도 | 반려 사유·이전 본문 미리보기 + 결재함 링크. |
| **승인 완료** | 지주 취합·보고서 반영 | `h-aggregate-write` 등 취합 화면으로 포커스(이미 정의한 딥링크와 정합). |

### 4.2 “제출 본문”의 출처

- 계열사가 DP에 대해 저장·제출한 **텍스트/버전**과 **결재 문서**가 1:1 또는 1:N일 수 있으므로, 백엔드에서 **“검토에 쓰는 스냅샷 id”**를 명시하는 것이 안전하다.  
- 프론트는 최소 **스니펫(예: 앞 500자)** + “전문은 결재함에서”를 기본으로 하고, 권한에 따라 전문 로드.

### 4.3 통합 결재함과의 관계

- **승인/반려의 최종 기록**은 **통합 결재함**과 맞추는 것을 전제로 한다.  
- 대시보드에서는 **미리보기 + 결재함 이동**이 최소이고, **즉시 승인/반려**를 넣을 경우에도 **같은 결재 엔진**을 호출해 이중 처리를 막는다.

---

## 5. 2번 영역: 페이지 현황 — 네모 카드 의존 탈피 · 실제 `h-write` 목록과 정합

### 5.1 단일 목차 소스

- 지주 **페이지별 작성** UI는 `HoldingPageByPageEditor`가 사용하는 **페이지 목록**(예: `holdingSrSds2024Pages.generated`에서 생성되는 목차)을 **정본(canonical)** 으로 삼는다.  
- 대시보드 “페이지 현황”은 **같은 `page`/`section`/`title` 키**로 조인해 **완료·작성중·미작성** 등 상태를 붙인다.  
- **별도의 임의 10개 카드**를 유지하지 않고, **목차가 늘어나면 대시보드도 같이 늘어나는** 구조가 목표다.

### 5.2 UI 형식(카드 필수 아님)

- **권장**: **섹션별 접기 테이블** 또는 **가상 스크롤 리스트**  
  - 컬럼 예: 페이지(범위), 제목, 공시기준 태그, 작성 상태, 최종 수정, 담당, **작업(페이지 작성으로 이동 / 미리보기)**.  
- 상단 KPI(전체/완료/작성중/미작성)는 **목차 목록에서 파생된 집계**와 항상 일치해야 한다.

### 5.3 상태 동기화

- `h-write`에서 저장·제출할 때마다 **섹션/페이지 단위 상태**가 서버에 반영되고, 대시보드는 **같은 API** 또는 캐시 무효화로 갱신한다.  
- 대시보드만의 로컬 “완료” 토글은 **실무와 어긋나므로**, 최종적으로는 **제거하거나 서버 반영 후에만** 표시한다.

### 5.4 딥링크(기존 유지·보강)

```text
/sr-report?mode=holding&holdingTab=h-write&sectionId={…}
```

- `sectionId`는 목차 노드 id와 1:1 (구현 시 `holdingPageData`와 동일 키 체계 사용).

---

## 6. 구현 단계(우선순위 재정렬)

1. **목차·DP 식별자 스키마 고정** — 매트릭스 셀·페이지 행이 같은 말을 하게 한다.  
2. **페이지 현황 UI를 목차 리스트 기반으로 교체** — 카드 그리드는 선택 사항.  
3. **매트릭스: 필터·피벗·가상 스크롤·드로어** 중 1~2개를 먼저 넣어 “DP 증가”에 대비.  
4. **셀 상세 API + 미리보기 + 독촉·결재 연동**.  
5. **집계 BFF** — KPI·히트맵·목차 요약을 일관되게.

---

## 7. 운영·품질 체크리스트

- [ ] DP·조직 수가 늘어나도 **스크롤·검색·필터** 없이 “한눈에 전부”를 요구하지 않는가  
- [ ] 제출 본문 미리보기와 결재 상태가 **동시에** 신뢰 가능한가  
- [ ] 미제출 독촉이 **이력·중복 발송 방지**와 함께 설계되었는가  
- [ ] 페이지 현황 숫자가 **`h-write` 목차**와 불일치하지 않는가  

---

## 8. 요약

- **DP 매트릭스**는 “전열 고정 표”가 아니라 **요약 + 탐색 + 셀 상세(본문·독촉·결재)** 로 재정의한다.  
- **제출**은 **문단 미리보기**를 전제로 **승인/반려(또는 결재함 연동)** 까지 한 흐름으로 설계한다.  
- **미제출**은 **독촉**을 1급 기능으로 둔다.  
- **페이지 현황**은 **실제 지주 SR 페이지 목록**과 동일 축으로 두고, **카드 레이아웃은 필수가 아니다** — 테이블·리스트가 확장에 유리하다.
