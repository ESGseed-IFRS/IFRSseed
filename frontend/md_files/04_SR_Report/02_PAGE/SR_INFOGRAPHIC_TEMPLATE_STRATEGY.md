# SR 페이지별 작성 — 인포그래픽 템플릿 전략

> 대상: 지주사 SR **페이지별 작성** (`HoldingPageByPageEditor` 등)  
> 목적: 인포그래픽 탭이 단순히 **표/차트 편집기로 이동하는 링크**에 그치지 않고, 스크린샷과 유사한 **고정 레이아웃 인포그래픽**을 추천·삽입·편집할 수 있게 하는 설계 방향을 정리한다.

---

## 1. 배경과 목표

### 1-1. 현재 상태

- **인포그래픽 추천** 탭은 공시기준 키워드(GRI 305 등)에 따라 **문구형 추천**을 보여 주고, 클릭 시 **그래프 생성기** 탭으로 전환하는 수준이다.
- 사용자 기대는 **Microsoft ESG 등 실제 보고서에 가까운 시각 템플릿**(원형 KPI, 타임라인, 피라미드, 아이콘+게이지 등)이다.

### 1-2. 목표

1. **추천 단계**에서 “표/그래프 만들기”가 아니라 **템플릿 카드**(썸네일 + 스타일 설명 + 참고 출처)가 노출된다.
2. 선택 시 해당 템플릿의 **미리보기**와 **필드 편집 UI**(수치, 연도, 라벨, 단위)가 제공된다.
3. 가능한 범위에서 **인포그래픽 내부 문구·수치**를 사용자가 수정하고, **페이지 본문 영역에 블록으로 삽입**한다.
4. (후순위) GHG 산정·그룹 통합 등 **앱 내 집계 데이터**를 초기값으로 주입한다.

---

## 2. 템플릿 민스샷(참고 UI)과 매핑

아래 네 가지는 **1차 템플릿 세트**로 정의한다. 이름은 제품 내 표기용이다.

| ID | 명칭(예시) | 역할 | 편집 가능 필드(예시) |
|----|------------|------|----------------------|
| `kpi-orbit` | KPI 원형 인포그래픽 | 중앙 목표(감축 %) + 위성 영역에 Scope1/2/3 | 중앙 %·라벨, 각 Scope별 수치·단위·짧은 설명, 색상 프리셋 |
| `reduction-timeline` | 감축 여정 타임라인 | 연도별 마커 + 배출량/단계 라벨 | 연도 목록, 각 점의 제목·수치·상태(기준년/진행/목표 등) |
| `scope-pyramid` | Scope 계층 피라미드 | 역삼각/층 구조로 Scope 비중 | Scope별 tCO₂e, 비율 표시 on/off, 층 순서(고정 vs 옵션) |
| `icon-kpi-row` | 아이콘 + 원형 KPI | 가로 배열의 소과제 KPI 게이지 | 항목 수(칸), 아이콘/제목, %, 부제(전년비 등) |

추가 템플릿은 **동일 스키마**(아래 3절)를 확장하는 방식으로만 추가한다.

---

## 3. 정보 구조(IA)와 데이터 모델

### 3-1. 템플릿 메타데이터(카탈로그)

각 템플릿은 코드에 하드코딩하거나 JSON 카탈로그로 관리한다.

- `templateId`, `title`, `description`
- `thumbnail` 또는 `previewComponent`
- `tags`: `["GRI 305", "GHG", "climate"]` 등 — **공시기준·토픽과 매칭**해 추천 순위에 사용
- `referenceReports`: `{ name, year }[]` — UI의 “참고 보고서” 영역
- `styleBullets`: `string[]` — 스타일 설명 불릿
- `schemaVersion`: 편집 데이터 마이그레이션용

### 3-2. 인스턴스 데이터(편집 상태)

페이지에 삽입되는 것은 “그림 파일”이 아니라 **`InfographicBlock`** 하나의 직렬화 객체다.

권장 필드:

```text
{
  "kind": "infographic",
  "templateId": "kpi-orbit",
  "schemaVersion": 1,
  "props": { ... 템플릿별 JSON ... },
  "dataSource": "manual" | "ghg_group_2025" | ...
}
```

- `props`는 **템플릿별 서브 스키마**로 TypeScript discriminated union 관리.
- `dataSource`가 `manual`이 아니면, **재동기화** 버튼으로 집계값을 다시 채울 수 있게 한다(덮어쓰기 전 확인).

### 3-3. 표/차트 블록과의 관계

- **ChartBlock / TableBlock**은 자유도가 높고 데이터 입력 중심.
- **InfographicBlock**은 **레이아웃 고정 + 제한된 필드**만 노출해 SR 편집자가 “보고서 톤”을 유지하기 쉽다.
- 한 페이지에 **둘 다** 허용하되, UI에서는 탭을 **「인포그래픽」 / 「차트(자유)」 / 「표」**로 명확히 분리하는 것이 좋다.

## 4. UX 전략

### 4-1. 인포그래픽 탭 흐름

1. **갤러리**: 공시기준·페이지 제목으로 정렬된 템플릿 카드(썸네일 + 제목 + 짧은 설명).
2. **선택 → 편집 패널**: 좌측 미리보기(SVG/React), 우측 또는 하단 **폼 필드**(숫자, 텍스트, 연도, 선택형).
3. **페이지에 추가**: 기존 “그래프를 페이지에 추가”와 동일하게 **본문 탭 하단 블록 목록**에 쌓인다.
4. **블록 내 편집**: 삽입 후에도 “편집”으로 동일 패널 재진입(모달 또는 인라인).

### 4-2. “그래프 탭으로 이동”의 위치

- 인포그래픽 탭에서는 **기본 동선을 갤러리 → 편집 → 삽입**으로 둔다.
- **고급 사용자**를 위해 “자유 차트로 전환” 링크만 보조적으로 둔다(현재 동작과의 연속성).

### 4-3. GHG 데이터 주입

- `dataSource: ghg_group_*` 선택 시, `GroupResults`·지주 GHG 대시보드 mock 등 **동일 숫자 체계**를 `props` 초기값으로 매핑.
- 매핑 테이블은 한 파일(`ghgToInfographicProps.ts` 등)에 모아 **단일 진실 공급원**으로 유지한다.

---

## 5. 기술 전략

### 5-1. 렌더링

- **SVG + React** 권장: 확대 시 선명, 인쇄/PDF 내보내기에 유리, 필드와 1:1 바인딩 용이.
- 복잡한 레이아웃은 **작은 서브 컴포넌트**로 분리(`KpiOrbitSvg`, `TimelineSvg`, …).

### 5-2. 편집 가능 범위

- **1단계**: 텍스트·숫자·연도·색상 프리셋.
- **2단계**: 항목 개수 변경(타임라인 포인트 수, KPI 칸 수) — 스키마에 `maxItems` 제한.
- **비권장(초기)**: 자유 곡선·픽셀 단위 드래그(유지보수·일관성 비용 큼).

### 5-3. 상태 저장

- 페이지별 `blocks` 배열에 `InfographicBlock` 포함 → 로컬 state → 이후 **저장 API / localStorage 초안**과 동일 정책.
- 스키마 버전 업 시 **마이그레이션 함수**로 구 블록 호환.

### 5-4. 접근성·국제화

- 의미 있는 라벨은 SVG `title` / `aria-label` 또는 숨김 텍스트.
- 단위(tCO₂e)는 `props`에 분리해 **표기만 변경** 가능하게.

## 6. 단계별 로드맵(제안)

| 단계 | 내용 |
|------|------|
| **P0** | 카탈로그 4종 + `InfographicBlock` 타입 + 갤러리 UI + 편집 폼 + 본문 삽입·미리보기 |
| **P1** | 삽입 후 블록 편집, GHG 그룹 수치에서 초기 주입, “자유 차트”와 탭 분리 카피 정리 |
| **P2** | PDF/HTML 내보내기 시 SVG 인라인 품질 검증, 추가 템플릿 2~3종 |
| **P3** | 브랜드 테마(색상 팔레트) 전사 설정 |

---

## 7. 리스크와 결정 사항

- **저작권/레퍼런스**: 카드에 인용하는 보고서명은 **디자인 참고** 수준이며, 실제 로고·문구는 고객 자산으로 교체 가능해야 한다.
- **성능**: 갤러리 썸네일은 정적 이미지 또는 경량 SVG; 무거운 프리뷰는 선택 후에만 마운트.
- **과도한 자유도**: 인포그래픽은 “**제한된 편집**”이 제품 가치다. 자유 시각화는 기존 차트/표에 맡긴다.

---

## 9. 요약

인포그래픽은 **“추천 문구 → 그래프 탭 이동”**이 아니라 **“템플릿 카탈로그 → 제한된 필드 편집 → SVG 블록 삽입”**으로 재정의한다.  
데이터는 **수동 입력을 기본**으로 하고, GHG·그룹 산정은 **선택적 시드**로 연결한다. 이렇게 하면 스크린샷과 같은 UX에 가깝게 가면서도, 구현 범위와 유지보수를 통제할 수 있다.
