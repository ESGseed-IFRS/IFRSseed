# 로그인·등록 — 반반 레이아웃 및 법인 인증 게이트 전략

작성일: 2026-03-26  
관련 문서: `LOGIN_REGISTER_IMPLEMENTATION_STRATEGY.md`  
대상 구현: `src/components/auth/AuthLoginScreen.tsx`, `src/components/auth/AuthRegisterScreen.tsx` (및 향후 공통 `AuthSplitLayout` 분리 시)

---

## 1. 목표 요약

1. **레이아웃**: 로그인·등록 화면 모두 **좌(브랜드)·우(폼) 영역이 뷰포트 기준 약 50% : 50%** 로 보이도록 조정한다.  
2. **등록 1단계**: **등록 코드** 입력이 아니라 **법인 아이디 + 패스워드**로 “법인(마스터) 자격”을 확인한 뒤, 동일 플로우로 **서브계정(담당자) 정보 입력 → 가입 완료**로 진행한다.

기존 전략 문서의 라우팅·브랜딩·3단계 위저드 구조는 유지하되, **1단계의 입력 항목·API·카피**만 본 문서 기준으로 갱신한다.

## 2. 현재 구조와 문제 인식

### 2-1. 레이아웃

- `AuthLoginScreen` / `AuthRegisterScreen` 공통 패턴:  
  - **좌**: `flex: 1` → 남는 공간을 모두 차지  
  - **우**: 고정 폭(`AuthLoginScreen` 약 480px, `AuthRegisterScreen` 약 520px) + `flexShrink: 0`  
- 결과: 가로가 넓은 화면에서 **좌측이 훨씬 넓어 보이는(대략 75/25 체감)** 비대칭이 된다.

### 2-2. 등록 플로우

- 1단계는 **등록 코드 검증** → 성공 시 법인 메타(`company` 등)를 세팅한 뒤 2·3단계 진행.  
- 요구사항은 **“법인 계정으로 로그인하듯 입력”** 으로 게이트를 통과시키는 방식으로 바뀐다.

## 3. 반반(50/50) 레이아웃 전략

### 3-1. 권장 CSS/Flex 규칙

- **루트**: `display: flex`, `minHeight: 100vh`, `width: 100%` 유지.  
- **좌·우 각각**:  
  - `flex: 1 1 50%` 또는 `width: 50%` + `minWidth: 0` (내부 오버플로 방지)  
  - 동일한 `flex-basis`로 **양쪽이 항상 동일 비율**을 갖게 한다.  
- **고정 폭 우측 패널 제거**: `width: 480` / `520`, `flexShrink: 0` 를 제거하고, 우측도 `flex: 1` 기반으로 맞춘다.  
- **카드**: 우측 내부의 폼 카드는 기존처럼 `maxWidth: 440` 등으로 **콘텐츠 폭만 제한**하고, 패널 자체는 50% 영역 안에서 가운데 정렬(`alignItems: 'center'`, `justifyContent: 'center'`)을 유지한다.

### 3-2. 반응형(선택·권장)

- **중단점 이하**(예: `max-width: 900px`):  
  - 세로 스택: `flexDirection: 'column'`  
  - 브랜드 패널 높이를 `minHeight` 또는 `auto`로 줄이고, 폼을 하단에 풀폭에 가깝게 배치.  
- 반반 비율은 **데스크톱 우선** 요구이므로, 모바일에서는 가독성이 우선이어도 된다.

### 3-3. 코드 정리 방향

- 두 화면의 `left` / `right` 스타일이 거의 동일하므로, 중복 제거 시 **`AuthSplitLayout`**(또는 공통 `S.splitLeft` / `S.splitRight`)로 한 번만 정의하는 것이 유지보수에 유리하다.  
- **범위**: 본 작업에서 레이아웃 토큰만 추출해도 되고, 별도 PR로 분리해도 된다.

## 4. 등록: 등록 코드 → 법인 아이디·패스워드 전략

### 4-1. UX 정의

| 항목 | 변경 전 | 변경 후 |
|------|---------|---------|
| 1단계 제목 | 등록 코드 확인 | 법인 계정 확인(가칭) |
| 입력 필드 | 등록 코드 1개 | 법인 아이디(로그인 ID) + 패스워드 |
| 성공 시 | 코드에 매핑된 `company` 등 | 인증 응답의 법인·세션 토큰 등(백엔드 스펙 따름) |
| 스테퍼 라벨 | `코드 확인` | `법인 확인` / `계정 확인` 등 짧은 2~4자어 |

좌측 히어로·플로우 가이드(01, 02, 03) 문구도 **“코드 입력” → “법인 계정으로 확인”** 으로 맞춘다.

### 4-2. 상태·검증 (프론트)

- 빈 값: “법인 아이디와 비밀번호를 모두 입력해주세요.” 등 **로그인과 동일한 톤** 권장.  
- 제출 중 스피너·에러 메시지는 기존 코드 스텝과 동일 패턴 유지.  
- **민감 정보**: 패스워드는 URL 쿼리·`localStorage`에 저장하지 않는다. 2단계로 넘길 때 필요한 것은 **서버가 내려준 등록용 토큰/세션** 뿐이어야 한다(아래 API).

### 4-3. 백엔드 API 방향 (가칭 — 스펙 확정 후 치환)

기존 전략의 `POST /api/auth/register/verify-code` 를 다음 중 하나로 대체·병행한다.

| 방안 | 설명 |
|------|------|
| **A. 전용 엔드포인트** | `POST /api/auth/register/verify-corporate` — body: `{ corporateLoginId, password }` → 성공 시 `{ company, registrationToken }` (또는 httpOnly 쿠키 세션) |
| **B. 로그인과 동일 + 스코프** | 법인 계정 로그인 API에 `intent: 'sub_register'` 등을 두어, 성공 응답만으로 서브 등록 위저드 진입 허용 |

- **2단계 제출**(`POST /api/auth/register`) 시: 기존 페이로드에 **`registrationToken`**(또는 쿠키 기반이면 생략)을 포함해 “이 세션이 법인 인증을 거쳤음”을 서버가 검증한다.  
- 보안: 짧은 TTL, 1회용 토큰, HTTPS 필수, 실패 메시지는 **계정 존재 여부 노출 최소화**.

### 4-4. 데모·목(Mock) 전환

- 기존 `CODE-2024` 데모는 제거하거나 개발 전용으로 대체.  
- 목 단계에서는 예: `corporateId` / `password` 조합 한 벌로 `company` 메타를 반환하도록 `AuthRegisterScreen` 내부 목만 교체한다.

### 4-5. 컴포넌트·심볼 이름 (선택)

- `RegisterCodeStep` → `RegisterCorporateGateStep` 등으로 리네이밍하면 의도가 분명해진다.  
- 아이콘: `KeyIcon` 대신 `UserIcon`+`LockIcon` 조합 등 로그인과 시각적 일관 유지.

## 5. 로그인 화면과의 정합성

- 로그인은 이미 **아이디·비밀번호**이므로 **레이아웃 50/50만 적용**하면 된다.  
- 등록 1단계와 **필드 구성이 유사**해지므로, 라벨·플레이스홀더·에러 문구를 통일하면 사용자 혼동이 줄어든다.  
- “법인 로그인”과 “서브 등록용 법인 확인”이 **동일 자격 증명**인지 제품 정책에 따라 문구를 구분한다(예: 등록 화면에 “서브계정 등록을 위해 법인 관리자 계정으로 확인합니다”).
