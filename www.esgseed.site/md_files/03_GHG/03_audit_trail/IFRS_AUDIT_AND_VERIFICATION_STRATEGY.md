# IFRS 감사 대응 및 GHG 검증의견서 전략

> **배경**: 온실가스 배출량 산정 결과 페이지의 "IFRS 감사대응"이 현재 "무엇을 해야 한다"는 안내 위주라서, **앞서 계산된 데이터가 IFRS·GHG Protocol 요건에 실제로 반영되었는지 확인**하고, **GHG Protocol 검증의견서 대응에 필요한 서류가 작성·다운로드**되도록 하는 전략을 정리한다.  
> **참조**: 본 문서는 **IFRS_RESULT_v2.md**(OCR 기반 디지털 증빙, Evidence 1:1 매핑, Scope별 증빙, 검증의견서 자동화, Auditor View) 내용을 반영한다.

## 2. 화면 구조 제안

### 2-1. IFRS 감사대응 탭 내부 구분

상단 탭 3개(산정 결과 | IFRS 감사대응 | 리포트 생성)는 **유지**하고, **IFRS 감사대응** 탭 안에서 다음 두 영역을 **서브 탭** 또는 **고정 섹션**으로 구분한다.

| 순서 | 서브 영역 | 표시 내용 |
|------|-----------|-----------|
| **1** | **요건 반영 확인** | IFRS S2·GHG Protocol 요구사항별로 "현재 산정 데이터가 해당 요건을 충족하는지" 체크리스트 + 상태(✓/△/✗) 표시. 클릭 시 어떤 데이터가 반영되었는지 요약 노출. |
| **2** | **검증의견서·서류 작성** | GHG Protocol / KMR 검증의견서 대응에 필요한 **서류 목록** + 각 서류 **자동 생성(미리보기)·다운로드** 버튼. |

- **UI**: 서브 탭 2개 — `요건 반영 확인` | `검증의견서·서류 작성` (직사각형, 날카로운 모서리, IFRS_RESULT.md §0-2 준수).
- **접기/펼치기**: 각 서브 탭 내용은 §2~§6 스타일로 **섹션 고정 표시**(접기/펼치기 없음 권장).

## 3. 요건 반영 확인 (데이터 ↔ 요건 매핑)

### 3-1. 목적

감사인·검증인이 "시스템에 입력된 데이터와 설정이 IFRS S2·GHG Protocol 요구사항을 만족하는지" 한 화면에서 확인할 수 있게 한다.

### 3-2. 요건 항목과 데이터 소스 매핑

아래 항목은 **체크리스트 한 행**으로 표시하고, **데이터 소스(스토어/계산 결과)**와 비교해 **상태**를 부여한다.

| 요건 분류 | 요건 항목 | 확인 방법 (데이터 소스) | 상태 판단 |
|-----------|-----------|--------------------------|-----------|
| **산출 근거** | 배출계수(EF)·GWP·산식 명시 | `boundaryPolicy.efDbVersion`, MethodologyDetailPanel 상수, Scope별 적용 산식 | EF·GWP·산식이 산정에 적용되어 있으면 ✓ |
| **산출 근거** | Scope별 산식 적용 (Scope 1/2 전력·열) | scope1/scope2 계산 로직, Step4Results 집계 | 실제 계산에 Scope별 산식 사용 여부 |
| **산출 근거** | 순발열량(NCV) 적용 기준 명시 | 배출계수 DB 또는 methodology 메타 | NCV 출처·버전 표시 여부 |
| **데이터 품질** | 활동자료 실측/추정 구분 (Data Quality) | `scope1.stationary/mobile`, `scope2.electricity` 각 행의 `dataQuality?.dataType` | 1건이라도 `dataType` 있으면 ✓, 전수 없으면 △ |
| **데이터 품질** | 불확실성 등급(High/Medium/Low) | Step4Results의 `confidenceLevel` (실측만 → High, 추정 포함 → Medium) | 계산 결과와 동일하게 표시 |
| **조직·운영 경계** | 조직 경계(지배권/운영통제) 명시 | `boundaryPolicy.organizationBoundary` | 값 존재 시 ✓ |
| **조직·운영 경계** | 보고 연도·산정 범위(Scope 1/2/3 포함 여부) | `boundaryPolicy.reportingYear`, `operationalBoundary.scope1Included`, scope2Included | 설정 존재 시 ✓ |
| **Scope 2** | 위치 기반·시장 기반 이중 산정 (IFRS S2) | `operationalBoundary.scope2Included`, Step4Results의 s2·s2Market | "동시 산정" 또는 시장 기반 값 계산 시 ✓ |
| **증빙·계보** | 수정 사유 기록 (감사 추적) | 셀 수정 시 사유 입력(CellEditModal 등) — 백엔드/로그 있으면 연동 | 로그 있으면 ✓, 없으면 "시스템 연동 후 표시" |
| **증빙·계보** | 증빙 파일(고지서·영수증) 연결 | scope1/scope2/scope3 내 첨부·receipts 필드 | 1건이라도 연결 시 △, 전수 권장 시 안내 |
| **OCR·디지털 증빙** | OCR 기반 1:1 매핑 (Evidence Viewer) | 좌측 PDF 뷰어+OCR 하이라이트, 우측 시스템 입력값, Manual vs OCR 대조 | OCR 연동 시 ✓, 미구현 시 △(준비 중) |
| **OCR·디지털 증빙** | OCR 추출값 vs 입력값 대조·Audit Mark | 일치 시 Verified(Green), 불일치 시 Flagged(Red)+사유 입력 | 대조 결과 저장 시 ✓ |
| **OCR·디지털 증빙** | OCR 신뢰도 점수(Confidence Score) | 95% 미만 시 수동 검토 알림 | 신뢰도 필드·알림 로직 있으면 ✓ |
| **OCR·디지털 증빙** | 증빙 추적 가능성(Traceability) | 배출량 레코드(Row)↔고지서/영수증 PDF 하이퍼링크, 뷰어 내 포커싱(Zoom-in) | 파일 연동·뷰어 연동 시 ✓ |
| **Scope별 증빙** | Scope 1 영수증-자산 매칭 | 유류 영수증 차량번호 ↔ 자산 마스터 매핑, 중복 영수증 필터링 | 영수증·자산 연동 시 ✓ |
| **Scope별 증빙** | Scope 2 고지서 1:1 매칭·Market-based | 한전 고지서 당월 사용량 매칭, REC/PPA 증서 연결 | 고지서·REC/PPA 증빙 연결 시 ✓ |
| **Scope별 증빙** | Scope 3 데이터 품질 등급(Tier1/2)·인보이스 | 실측+OCR 완료=Tier1, Spend-based=Tier2, ERP 전표↔인보이스 PDF | dataQuality·receipts 연동 시 ✓ |
| **Auditor View** | 전체 증빙 확보율(%) | 전체 배출량 중 OCR/디지털 증빙 완료 비율 | 집계 API/로직 있으면 ✓ |
| **Auditor View** | 미비 항목 리포트 | 증빙 누락·OCR 신뢰도 낮은 항목 자동 리스트 | 리포트 생성 시 ✓ |

### 3-3. 화면 표시 방식

- **테이블**: 요건 항목 | 요건 설명(한 줄) | 상태(✓ 충족 / △ 일부 충족 / ✗ 미충족 / − 해당 없음) | "상세" 링크.
- **상세**: 해당 행 클릭 시 "어떤 데이터가 이 요건을 채우는지" 요약(예: "적용 배출계수: 환경부 2025, GWP AR5 100년").
- **데이터 소스**: `useGHGStore`에서 boundaryPolicy, scope1, scope2, scope3, factorYear 등 구독 후 위 로직으로 상태 계산.

### 3-4. 구현 시 유의

- 백엔드에 lineage·audit 로그가 없으면 "수정 이력·승인 로그" 항목은 **미연동** 상태로 표시하고, 플레이스홀더 문구로 "시스템 연동 후 표시" 처리.
- 증빙 파일·OCR 연동이 없으면 "OCR 기반 1:1 매핑"·"Auditor View" 항목은 **△**(준비 중)으로 두고, §5 모듈 구현 순서에 따라 단계 반영.

## 4. 검증의견서·서류 작성 (GHG Protocol / KMR 대응)

### 4-1. 목적

GHG Protocol 기반 **온실가스 검증의견서** 및 KMR 양식 대응에 필요한 **서류를 목록화**하고, **현재 산정 데이터로 자동 생성·다운로드**할 수 있게 한다.

### 4-2. 필요 서류 목록 (검증의견서 대응)

IFRS_RESULT_v2.md §4 검증의견서 자동화 및 실무 관행을 반영한 항목:

| 순서 | 서류명 | 내용 | 자동 생성 가능 여부 |
|------|--------|------|---------------------|
| 1 | **검증 범위 요약** | 등록 사업장·배출원 목록, 산정 연도, 조직 경계, Scope 1/2/3 포함 여부 | ✓ boundaryPolicy, facilities, scope1/2/3 에너지원 목록으로 생성 |
| 2 | **배출량 집계표** | Scope 1/2/3 합계 및 카테고리별(에너지원별) 상세 tCO₂e | ✓ Step4Results와 동일 집계 + scope별 상세 테이블 |
| 3 | **데이터 품질 통계** | 실측 vs 추정 비율, **실측 증빙(OCR 완료) 비중**(%), 데이터 품질 등급 요약 (v2 §4) | ✓ dataQuality 집계, OCR 완료 건수·증빙 연결 건수 반영 |
| 4 | **적용 배출계수·산식 요약** | EF 출처·버전, GWP, Scope별 산식 문구 | ✓ methodology 상수 + boundaryPolicy |
| 5 | **활동자료 요약 (Activity Data Summary)** | 검증 대상 기간 총 사용량(연료/전력 등), 단위별 합계 | ✓ scope1/2/3 amount 집계 |
| 6 | **증빙 목록(선택)** | 고지서·영수증 첨부 목록(파일명, 연계 항목) | ✓ scope3 receipts 등 기존 evidence payload 확장 |

### 4-3. 자동 생성 방식

- **데이터 소스**: 기존 `buildEvidencePayload`(boundaryPolicy, scope1, scope2, scope3) 확장 또는 동일 payload 활용.
- **출력 형식**:
  - **PDF**: 검증 범위 요약 + 배출량 집계표 + 데이터 품질 통계 + 적용 계수·산식 (EvidencePdfDocument 스타일 확장 또는 전용 "검증의견서 부속 서류" PDF 컴포넌트).
  - **Excel**: 시트별로 "검증 범위", "배출량 집계표", "데이터 품질", "활동자료 요약", "적용 계수" 시트 구성 (기존 evidenceReport Excel 다운로드 확장).
- **파일명 예시**: `ghg-verification-support-YYYY-MM-DD.pdf`, `ghg-verification-support-YYYY-MM-DD.xlsx`.

### 4-4. 화면 표시 (검증의견서·서류 작성 탭)

- **서류 목록 카드**: 위 1~6 항목을 카드 또는 테이블로 나열. 각 행에 "미리보기"(해당 섹션만 보기), "다운로드(PDF/Excel 포함)" 버튼.
- **통합 다운로드**: "검증의견서 부속 서류 일괄 다운로드" 버튼 → PDF 1부 + Excel 1부 생성.
- **안내 문구**: "아래 서류는 GHG Protocol·KMR 검증의견서 제출 시 참고용으로 활용할 수 있습니다. 검증기관별 추가 요청이 있을 수 있으니 확인 후 제출하세요."

### 4-5. 기존 Step5 리포트와의 관계

- **Step5 (리포트 생성 탭)**: 기존 Evidence Report(산정설정·Scope1·2·3·영수증 목록) — **유지**.
- **검증의견서·서류 작성**: "검증의견서 대응"에 특화된 **추가 출력물** — 검증 범위·집계표·데이터 품질 통계 등 **검증인 제출용 구조**로 정리.  
  - 구현 시 Step5의 `buildEvidencePayload`·`EvidencePdfDocument`·Excel 유틸을 **재사용·확장**하여 중복을 줄인다.

## 5. OCR·디지털 증빙 모듈 (IFRS_RESULT_v2.md 반영)

v2의 **Digital Evidence 1:1 매핑**, **Scope별 증빙**, **검증의견서 자동화**, **Auditor View**를 전략에 명시적으로 포함한다. 구현 시 "요건 반영 확인" 체크리스트(§3-2) 및 "검증의견서·서류 작성"과 연동한다.

### 5.1. OCR 기반 1:1 매핑 (Evidence Viewer)

- **화면 구성** (v2 §2)
  - **좌측 (Evidence Viewer)**: 업로드된 PDF(고지서, 영수증, 인보이스) 뷰어 + **AI OCR 하이라이트**.
  - **우측 (System Data)**: 시스템에 입력된 활동도 데이터 및 계수 정보.
- **OCR 자동 추출 및 대조**
  - 사용량, 기간, 고객번호 등 핵심 수치를 인식하여 **시스템 입력값과 실시간 대조**.
  - **신뢰도 점수(Confidence Score)**: 인식 정확도 표시, **95% 미만 시 수동 검토 알림**.
- **대조 로직 (전력 고지서 예시, v2 §1)**
  - 추출 항목: 한전 고지서 내 [사용기간, 당월사용량(kWh), 기본요금, 전력량요금].
  - `Manual_Input_Value` vs `OCR_Extracted_Value` → 일치 시 **Status 'Verified' (Green)**, 불일치 시 **Status 'Flagged' (Red)** 및 **사유 입력 강제**.
- **증빙 자료 추적 가능성 (Traceability, v2 §2)**
  - 각 배출량 레코드(Row)에 해당 월 고지서 원본 **PDF 하이퍼링크 매핑**.
  - PDF 뷰어 내 추출 데이터 위치 **자동 포커싱(Zoom-in)**.
- **수정 이력(Audit Trail)**: 수동 수정 시 수정 전/후 값·사유를 타임스탬프와 함께 기록 (v2 §2).

### 5.2. Scope별 증빙 확장 (v2 §3, §Scope 1/3)

| Scope | 내용 | 구현 포인트 |
|-------|------|-------------|
| **Scope 1** | 영수증-자산 매칭 | 유류 영수증 내 **차량번호** 인식 → 자산 마스터 ID 자동 매핑. 동일 날짜/차량 **중복 영수증 필터링**. 산출 근거(수량×NCV×EF) 시각화. |
| **Scope 2** | 고지서 매핑·Market-based | 한전 고지서 **당월 사용량**과 시스템 값 1:1 매칭. **REC 구매 증서, PPA 계약서** 등 디지털 증빙 연결. |
| **Scope 3** | 데이터 품질 등급·인보이스 | **Tier 1 (실측)**: 협력사 고지서/영수증 OCR 매핑 완료. **Tier 2 (추정)**: Spend-based. 구매 시스템(ERP) 전표 ↔ 외부 인보이스 PDF 1:1 연결. |

### 5.3. 통합 대시보드 (Auditor View, v2 §3)

- **전체 증빙 확보율**: 전체 배출량 중 **OCR/디지털 증빙이 완료된 비율(%)** 표시.
- **미비 항목 리포트**: 증빙 누락 또는 **OCR 신뢰도가 낮은 항목** 자동 추출 리스트.

이 섹션(§5)은 **요건 반영 확인**에서 "OCR·디지털 증빙", "Scope별 증빙", "Auditor View" 항목의 데이터 소스·판단 기준이 되며, 단계적 구현 시 1) 증빙 파일 연결 → 2) OCR 추출·대조 API 연동 → 3) Evidence Viewer UI → 4) Auditor View 순으로 확장할 수 있다.

## 8. 요약

- **IFRS 감사대응** 탭을 "안내만 있는 페이지"에서 **계산 데이터가 요건을 충족하는지 확인하는 페이지**로 바꾼다.
- 탭 내부를 **요건 반영 확인**과 **검증의견서·서류 작성** 두 영역으로 나누고,  
  - **요건 반영 확인**: IFRS S2·GHG Protocol 요구사항 + **IFRS_RESULT_v2의 OCR 1:1 매핑·Scope별 증빙·Auditor View** ↔ 스토어/산정 결과 매핑 후 ✓/△/✗ 표시.  
  - **검증의견서·서류 작성**: 검증 범위 요약, 배출량 집계표, **실측 증빙(OCR 완료) 비중을 포함한** 데이터 품질 통계 등 필요 서류를 자동 생성·다운로드.
- **OCR 기반 디지털 증빙**(Evidence Viewer, 대조 로직, Traceability, Scope별 영수증/고지서/인보이스, 통합 대시보드)은 §5에 명시하며, 구현 단계 7~9에서 단계적으로 반영한다.
- 기존 **산정 결과 / 리포트 생성** 탭과 **Step5 Evidence Report**는 유지하며, 검증 서류는 Evidence payload와 PDF/Excel 생성을 확장해 구현한다.
