```bash
cd backend
```


## 예상 질문 및 답변

### Q1: "계열사가 데이터를 조작하면 어떻게 하나요?"
**A:** 지주사가 승인 전 검토하며, 이상치 감지 시스템(`ghg_anomaly_scan_results`)이 자동으로 플래그합니다. 또한 전년 대비 큰 변동이 있으면 경고가 표시됩니다.

### Q2: "실제 기업에서 이렇게 운영하나요?"
**A:** 네, 대부분의 대기업 그룹이 유사한 방식을 사용합니다. 계열사가 자체 산정 후 지주사에 제출하고, 지주사가 통합 보고서를 작성합니다. ISO 14064-1, GHG Protocol에서 권장하는 방식입니다.

### Q3: "Scope 3는 왜 계열사도 산정하나요?"
**A:** IFRS S2에서는 기업의 가치사슬 전체 배출량을 요구합니다. 계열사도 자체 공급망(Cat.1), 출장(Cat.6), 통근(Cat.7) 등 Scope 3를 산정하고, 지주사는 투자(Cat.15)로 계열사 배출량을 포함합니다.

### Q4: "데이터 출처 표시가 왜 중요한가요?"
**A:** IFRS S2와 ESRS는 데이터 출처 및 검증 수준 공시를 의무화합니다. 투자자들이 데이터 신뢰성을 판단하는 핵심 정보입니다.

## 기술 스택 어필 포인트

### 1. 데이터 아키텍처
```
계열사 (EMS/ERP/HR/EHS/MDG)
    ↓ CSV Upload
staging_*_data (PostgreSQL)
    ↓ GHG Calculation Engine
ghg_emission_results
    ↓ API: /subsidiary/submit
subsidiary_data_submissions (status: submitted)
    ↓ 지주사 승인
subsidiary_data_submissions (status: approved)
    ↓ GroupAggregationService
SR 보고서 (데이터 출처 표시)
```

### 2. 핵심 기술
- **Backend**: FastAPI + PostgreSQL + SQLAlchemy
- **Frontend**: Next.js + React + Zustand
- **GHG 산정**: pandas + emission factor DB
- **AI 생성**: LangGraph + Gemini/GPT
- **데이터 추적**: source_company_id (모든 테이블)

### 3. 차별화 요소
- ✓ **실제 배출량 사용** (삼성SDS 2024 검증값)
- ✓ **End-to-End 자동화** (수집 → 산정 → 제출 → 승인 → 보고)
- ✓ **데이터 출처 투명성** (모든 수치에 출처 표시)
