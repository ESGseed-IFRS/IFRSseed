# 삼성SDS 언론·보도 수집 파이프라인 — LLM 컨텍스트·프롬프트 팩

이 파일은 **[전체 설계·구현 스펙 `EXTERNAL_COMPANY_DATA_SAMSUNG_SDS_NEWS.md`](EXTERNAL_COMPANY_DATA_SAMSUNG_SDS_NEWS.md)** 를 **대체하지 않습니다.**  
LLM에게 **짧은 사실·역할·출력 형식(프롬프트)** 을 주고, **세부 로직·표·DOM·마이그레이션**은 **반드시 스펙 문서 본문**을 같은 대화에 붙여 넣어야 합니다.

---

## LLM에게 줄 때 권장 패키지 (중요)

| 구성 요소 | 파일 | 역할 |
|-----------|------|------|
| **1) 스펙 본문** | `EXTERNAL_COMPANY_DATA_SAMSUNG_SDS_NEWS.md` | §1~§13 전체 — 로직·표·API·DB·Alembic 근거 |
| **2) 이 브리프** | `EXTERNAL_COMPANY_DATA_SAMSUNG_SDS_NEWS_LLM_BRIEF.md` | 용어·**레이어·플로우**·숫자 요약 + **아래 부록 프롬프트** |
| **3) (선택) 소스** | 스펙 §7 트리의 `.py` 일부 | LLM이 **클래스·함수 단위로** 설명해야 할 때만 추가 첨부 |

**붙이는 순서 (권장)**

- **비개발자 요약**: 부록 A 또는 B → 브리프(또는 스펙 §1.5).
- **개발자 + 스펙 로직만**: 부록 C 또는 D → **스펙 전문**.
- **개발자 + 플로우·레이어·기술 스택까지 (제미나이 등)**: **부록 E 또는 F** → 스펙 전문 → **이 브리프의 「아키텍처·플로우」절** (코드 첨부는 선택).

**주의**

- 상대 경로 링크만 있는 외부 문서는 LLM이 열 수 없다. 필요한 본문은 **같이 붙인다.**
- 소스를 첨부하지 않았으면 **파일명·시그니처 추측 금지**. 스펙·브리프·첨부 코드에 있는 것만 말한다.

## 아키텍처·플로우 (개발자용 요약)

이 절은 스펙 **§7 코드 구조**와 맞춘 **역할 분담**이다. 세부 파일명은 §7 트리를 따른다.

### 한 줄 플로우 (요청 → 저장)

**진입(FastAPI)** → **오케스트레이터**(앵커 UUID 해석) → **인제스트 서비스**(0~N단계 조율) → **fetch / parse / (임베딩)** → **레포지토리**(upsert·상태 저장) → **PostgreSQL**.

### 레이어별 역할

| 레이어 | 역할 | 이 파이프라인에서 하는 일 |
|--------|------|---------------------------|
| **API 라우터** (`external_company_router` 등) | HTTP 경계, 요청/응답 스키마 | `POST .../sds-news/ingest`(수집 한 번 실행), `GET .../ingest-state`(**`ingest_state` 행 진단·ETag·`last_list_source` 확인**). **Pydantic 모델**로 JSON 검증·직렬화(문서상 **API DTO**에 해당). |
| **앱 진입·폴링** (`data_integration/main.py` 등) | 프로세스 수명, 백그라운드 tick | 주기마다 오케스트레이터와 동일 ingest 호출. **이전 실행 미종료 시 스킵**. |
| **오케스트레이터** | 유스케이스 얇은 조립 | 환경의 앵커 회사 ID 해석 후 **인제스트 서비스**에 위임. 도메인 규칙의 대부분은 서비스에 있다. |
| **도메인 서비스** | 한 번의 ingest 시퀀스 | HEAD/ETag·목록·상한·2·3단계 병렬·행 매핑·임베딩 호출·upsert 루프·**`ingest_state`(변경 감지용 DB 상태 행)** 갱신 순서를 **한 곳에서** 조율. |
| **fetch 모듈** | I/O 경계 | `httpx` 동기 클라이언트로 GET/HEAD, 재시도·UA. **순수에 가깝게** HTML/JSON 바이트 수준. |
| **parse 모듈** | HTML/JSON → 구조화 | `beautifulsoup4`+`lxml` 등으로 DOM에서 필드 추출. **외부 언론**은 범용 셀렉터 위주. |
| **임베딩 보조** | 배치 벡터화 | 공통 **BGE-M3(FlagEmbedding 경로)** 로 제목·본문 리스트를 잘라 넣고, 행 dict에 벡터 필드 채움. 실패 시 로그만. |
| **레포지토리** | 영속성 어댑터 | **`external_company_data`**: 수집된 기사·메타를 키 기준 upsert. **`ingest_state`**: 작업별(`task_key`)로 “마지막으로 본 ETag·목록을 어디서 가져왔는지”만 짧게 저장·조회. **짧은 세션**·(상태용) **전용 엔진** 패턴은 스펙 §1.5. |
| **ORM 엔티티** (`models/bases`) | DB 테이블 매핑 | `ExternalCompanyData`(적재 본문), `IngestState`(**변경 감지 스냅샷 1행/작업**) — **JPA식 Entity**에 해당. |
| **상태·결과 타입** (`models/states`) | 애플리케이션 내부 전달 | `ParsedNewsItem`, `IngestResult` 등 — **도메인 DTO/값 객체**에 가깝고, API 스키마와는 별도. |

**VO vs DTO**: 이 코드베이스는 엄격한 VO/DTO 네이밍을 쓰지 않는다. **API 경계 = Pydantic**, **크롤 중간 결과 = dataclass 등 states**, **테이블 = SQLAlchemy 모델**로 이해하면 된다.

### 기술 스택 (구현 관점)

| 영역 | 기술 |
|------|------|
| API | **FastAPI**, **Pydantic** |
| HTTP | **httpx** (동기, follow_redirects, timeout) |
| HTML | **BeautifulSoup**, **lxml** |
| DB | **SQLAlchemy** ORM, **Alembic**, **PostgreSQL**, **pgvector** |
| 병렬 | **ThreadPoolExecutor** (2·3단계 I/O) |
| 임베딩 | **FlagEmbedding** / **BGE-M3** (`EmbeddingService` 등 플랫폼 공통) |
| 설정 | **`backend/core/config/settings.py`**, 환경변수 |

### 시퀀스(개발자용 단계 번호)

0. **변경 감지**: 삼성SDS **목록 HTML 페이지**(통상 index 계열 URL)와 **JSON 피드 `news.txt`** URL에 **각각 HTTP HEAD**만 보내 ETag를 얻는다 → DB의 **`ingest_state` 행**(직전 실행에서 저장한 ETag·소스)과 **`should_fetch`** 로직으로 비교 → 바뀌지 않았으면 **early return**(본문 GET 생략).  
1. **목록**: 위 HTML을 파싱해 링크 목록을 우선 구성하고, 실패하거나 목록이 비면 **`news.txt`**(JSON)로 동일 역할을 대체한다.  
1.5. **상한**: `max_items` / 설정.  
2·3. **항목별**: 1차 상세 GET → 외부 URL·메타 → 외부 기사 GET → 본문 파싱(병렬·동시성 상한).  
4. **행 준비**: `ParsedNewsItem` → upsert용 dict.  
5. **임베딩(옵)**: 설정 켜짐이면 배치로 벡터 채움.  
6. **저장**: 레포지토리 upsert 루프.  
7. **상태 커밋**: **`ingest_state`** 테이블에 이번에 실제로 쓴 목록 소스에 맞는 ETag·**`last_list_source`**(`html` / `news_txt`) 등을 저장해 다음 폴링이 비교할 수 있게 한다.

## 한 줄로 (파이프라인 요약)

정해진 주기(또는 API)마다 삼성SDS 뉴스 **목록(HTML 또는 `news.txt`) → 1차 상세 → 외부 언론 본문**을 거쳐 **기사 적재 테이블 `external_company_data`**에 upsert하고, **작업별 상태 테이블 `ingest_state`**에 마지막 ETag·목록 소스를 남겨 **다음 번에 목록이 바뀌었는지만** 빠르게 판단한다. SR 생성 시점에는 **DB 조회만** 전제한다.

## 비유 (초보자용)

- 뉴스 사이트는 **게시판**, 프로그램은 링크를 따라 **본문을 우리 DB 공책**에 정리한다.
- **이미지 파일 저장**은 스펙 범위 밖. **텍스트·URL** 중심.

## 꼭 알아 둘 용어

아래 **index HTML**·**`news.txt`**·**`ingest_state`** 는 서로 다른 리소스다. 문서 어디서든 헷갈리면 이 표를 먼저 본다.

| 용어 | 한눈에 보는 뜻 | 조금 더 설명 |
|------|----------------|--------------|
| **index(목록 HTML)** | 삼성SDS 뉴스 **목록이 그려지는 웹 페이지**(링크·썸네일 DOM). | 실제 URL은 스펙 §2·§6에 있으며, 여기서 링크를 뽑아 “무슨 기사가 있는지” 알 수 있다. |
| **`news.txt`** | 같은 목록을 **JSON으로 내려주는 피드 파일**. | HTML 파싱이 실패하거나 목록이 비면 **폴백**으로 사용한다. HEAD ETag도 따로 존재해 **HTML과 독립**으로 바뀔 수 있다. |
| **index·`news.txt` 이중 HEAD** | 목록이 **두 갈래**(HTML / JSON)라서, 변경 감지를 위해 **URL 두 곳에 각각 HEAD**를 친다는 뜻. | DB에는 마지막으로 **어느 갈래로 목록을 가져왔는지**(`last_list_source`)와 그때의 ETag가 저장된다. |
| **ingest** | 한 번의 **수집 실행**(API 한 번 또는 폴링 tick 한 번). | 그 안에서 HEAD → (필요 시) GET 목록 → 상세·외부 본문 → DB 저장까지 이어진다. |
| **1·2·3단계** | 파이프라인의 **본문 GET 단계**를 세는 말. | **1단계**: 목록(HTML 우선, 없으면 `news.txt`). **2단계**: 삼성SDS 상세 페이지. **3단계**: 외부 언론 기사 본문. *(0단계는 HEAD만으로 하는 변경 감지.)* |
| **`external_company_data`** | **수집 결과가 쌓이는 본 테이블**(기사·URL·본문·임베딩 등). | upsert 키는 문서 기준 `(anchor_company_id, source_url)`. |
| **`ingest_state`** | **“다음에 목록을 다시 받아야 하나?”만** 위한 **작업별 상태 테이블**(행 수 적음). | `task_key`(예: `sds_news_list`)당 대략 1행. **마지막 ETag**, **`last_list_source`**, 수정 시각 등. 본문 데이터는 여기 없다. |
| **`last_list_source`** | 직전 성공 실행에서 목록을 **HTML에서 봤는지 / `news.txt`에서 봤는지** 기록. | 값: `html` 또는 `news_txt`. 다음 HEAD 비교 시 **같은 갈래의 ETag**와 맞춰 본다(마이그레이션 039). 옛 행은 NULL일 수 있어 **두 HEAD 모두**와 정규화 비교하는 레거시 분기가 있다. |
| **ETag 정규화** | GET이 `"…"`만 주고 HEAD가 `W/"…"`만 줄 때도 **같은 태그로 본다**. | 저장·비교 시 weak 접두·따옴표 차이로 **불필요한 재수집**이 나지 않도록 코드에서 처리한다. |
| **임베딩** | 벡터 검색용 필드 채우기. | `category_embedding`(제목 우선, 없으면 category), `body_embedding`(본문). BGE-M3, 배치·길이 상한은 §8. |

## Neon·장시간 크롤과 DB 세션 (요지)

크롤이 길 때 **한 세션을 길게 묶지 않는다.** **`external_company_data`**(많은 행 upsert)는 짧은 세션으로 끊고, **`ingest_state`**(가벼운 1행 갱신)는 **전용 엔진·짧은 세션·pre_ping·pool_recycle**로 유휴 SSL 끊김을 완화. 스펙 §1.5·문서 본문 참고.

**스키마**: ORM에 **`last_list_source`**(목록을 HTML/`news.txt` 중 어디서 봤는지) 컬럼이 있으면 DB에 **Alembic 039** 적용이 필요하다. 미적용 시 ETag·HEAD 비교 단계에서 SQL 오류가 날 수 있다.

## 부록 A — 다른 LLM에게 붙이는 지시문 (초보자·비개발자 설명)

**사용법**: 아래 블록을 유저 메시지 **앞**에 두고, 이어서 **이 브리프 + (선택) 스펙 §1.5** 를 붙인다.

```text
역할: 너는 기술 문서를 바탕으로 비개발자·초보자에게 설명하는 튜터다.

규칙:
- 아래에 이어 붙인 Markdown만 근거로 답한다. 추측·외부 지식으로 내용을 채우지 않는다.
- 문서에 없으면 "문서에 없음"이라고 한다.
- 문서 안의 상대경로 링크는 열 수 없다. 링크를 전제로 한 추론은 하지 않는다.

출력 구조 (한국어, 완전한 문장):
1) 비유(게시판·링크 따라가기·우리 DB)로 전체를 5~8문장.
2) 3단계(목록 → 삼성SDS 상세 → 언론사 본문)를 단계마다 2~4문장.
3) 왜 배치인지, **목록 HTML·`news.txt` HEAD로 얻은 ETag**가 DB **`ingest_state`** 와 같으면 왜 본문 GET을 생략할 수 있는지 각 2~3문장.
4) DB에 무엇이 들어가는지 비전문가용 목록 또는 표. category_embedding=제목(또는 카테고리) 임베딩, body_embedding=본문 임베딩임을 명시.
5) 임베딩 기본 켜짐·끄는 환경변수 이름·본문 길이 상한·배치 크기(문서 기본값만).
6) 운영 숫자 표: 폴링 주기, HTTP 동시성, 목록 상한 0의 의미.
7) Neon 유사 환경에서 긴 유휴 DB 연결이 끊기는 문제와, 문서에 나온 대응을 비유 1~2문장 포함해 짧게.
8) DOM 표·셀렉터 나열은 하지 말고, 필요하면 "HTML에서 본문 블록을 찾는다" 한 줄 수준만.

마지막에 한 문장: 빠진 주제가 있으면 질문해 달라고 안내한다.
```

## 부록 C — 다른 LLM에게 붙이는 지시문 (개발자용·스펙 로직 심층 설명)

**사용법 (필수)**: 아래 블록을 **시스템 또는 유저 메시지 맨 앞**에 둔 뒤, **반드시 `EXTERNAL_COMPANY_DATA_SAMSUNG_SDS_NEWS.md` 전체**를 이어서 붙인다. 이 브리프의 표·용어는 **보조**이며, **모순 시 스펙 문서가 우선**이다.

```text
역할: 너는 백엔드/데이터 파이프라인을 동료 개발자에게 설명하는 시니어 엔지니어다.

입력: 바로 아래에 이어 붙인 파일은 저장소 문서 `EXTERNAL_COMPANY_DATA_SAMSUNG_SDS_NEWS.md`(및 선택적으로 요약 브리프)다. 레포 소스 코드는 첨부되지 않았을 수 있다. 첨부된 Markdown에만 근거해 답한다. 없는 내용은 추측하지 말고 "문서에 없음"이라고 한다. 상대경로로만 적힌 다른 파일은 열 수 없다.

과제: 첨부 스펙의 **구현·운영 로직을 개발자 관점에서 세세하게** 한국어로 설명한다. 다음을 빠짐없이 다룬다(해당 절이 문서에 없으면 "문서에 없음").

0) **레이어·플로우(요약)**: 스펙 §7 트리 기준으로 API → 오케스트레이터 → 서비스 → fetch/parse → 레포지토리 → ORM 테이블 순의 **책임 분리**를 1개 절로 요약한다. (브리프 `아키텍처·플로우` 절과 정합되게, 단 스펙에 없는 세부는 넣지 않는다.)
1) **목적·경계**: §1 — SR 경로와 배치 경로의 분리, 범위 제외(이미지 등).
2) **엔드투엔드 실행 순서**: §1.5 및 §7.2 수준으로, 0단계(HEAD/ETag)·1단계(HTML vs `news.txt`)·상한·2·3단계 병렬·upsert·임베딩·**`ingest_state`에 상태 저장** 순서를 **의사결정 분기**와 함께 서술.
3) **변경 감지·ETag**: §5 및 §12·§9.3 — **`ingest_state` 테이블**, `task_key`, **목록 HTML URL·`news.txt` URL에 대한 이중 HEAD**, **`last_list_source`**, 레거시 행(NULL 소스: 어느 갈래 ETag를 저장했는지 불명일 때 두 HEAD와 비교) 시 동작, GET vs HEAD ETag 표기 차이(weak/따옴표) 가능성.
4) **DOM·파싱**: §2·§3 — `#bThumbs`/`#sThumbs`(목록 HTML), **`news.txt` JSON 폴백**, 1차 상세 `p.txt a[href]`, 외부 본문 범용 선택자·난이도.
5) **DB 매핑**: §4 — `external_company_data` 컬럼과 소스 타입(`press`/`news`), upsert 키 정책 문구.
6) **HTTP·동시성**: §6·§7.3 — 재시도, 동시성, 폴링과 "이전 실행 미종료 시 스킵".
7) **환경변수·기본값**: §8 표 전체를 요약 표로 재정리(누락 금지).

9) **임베딩**: §1.5·§8·브리프 — 모델명, 배치, 본문 truncate, 실패 시 정책, 컬럼 매핑(`category_embedding` 명명 주의).
```

## 부록 E — 플로우·레이어·기술 스택 설명 (개발자 / 제미나이 등)

**목적**: 비유·쉬운 말 대신 **시스템 구성**, **데이터가 레이어를 어떻게 지나가는지**, **어떤 라이브러리가 어디 쓰이는지**, **오케스트레이터·서비스·레포·엔티티·API DTO** 역할을 말하게 한다.

**사용법**

1. 아래 블록을 **시스템 또는 유저 메시지 맨 앞**에 둔다.  
2. 이어 **`EXTERNAL_COMPANY_DATA_SAMSUNG_SDS_NEWS.md` 전체**를 붙인다.  
3. 이어 **이 브리프에서 「아키텍처·플로우」절부터 「운영 숫자」까지**(또는 브리프 전체)를 붙인다.  
4. (선택) 스펙 §7에 나온 **특정 `.py` 파일**을 추가 첨부하면, 그 파일 내용까지 근거로 **호출 관계**를 적게 할 수 있다.

```text
역할: 너는 백엔드 아키텍처를 동료 개발자에게 설명하는 시니어 엔지니어다. 비유·마케팅 톤은 쓰지 말고, 기술 용어와 플로우 중심으로 쓴다.

입력: 이어 붙인 Markdown은 (1) 스펙 문서 EXTERNAL_COMPANY_DATA_SAMSUNG_SDS_NEWS.md (2) 선택적으로 LLM_BRIEF의 아키텍처·플로우 절 (3) 선택적으로 일부 소스 파일이다. 첨부에 없는 파일·클래스·함수 이름은 추측하지 말고 "첨부에 없음"이라고 한다.

과제: 다음 순서로 한국어로 설명한다.

1) **엔드투엔드 플로우**: HTTP 요청(또는 폴링 tick)이 들어온 뒤 DB에 커밋되기까지의 **단계 번호(0~7)** 와 각 단계의 **입력/출력 데이터 형태**(문서·브리프에 기술된 수준만).
2) **레이어 다이어그램(텍스트)**: Router → Orchestrator → Ingest Service → (Fetch / Parse / Embedding helper) → Repository → DB 를 한 블록 ASCII 또는 불릿 트리로 그린다.
3) **역할 표**: 각 레이어(오케스트레이터, 도메인 서비스, 레포지토리, ORM 엔티티, API Pydantic 모델, models/states dataclass)가 **무엇을 알고 무엇을 모르는지** 한 줄씩.
4) **기술 스택 매핑**: FastAPI, httpx, BeautifulSoup, lxml, SQLAlchemy, Alembic, PostgreSQL/pgvector, ThreadPoolExecutor, BGE-M3/FlagEmbedding 등이 **어느 레이어에서** 쓰이는지 표로 정리한다.
5) **DTO/VO/Entity 구분**: 이 코드베이스에서는 엄격한 VO 용어를 쓰지 않을 수 있음을 밝히고, **API 경계(Pydantic) vs 메모리 내 파싱 결과(states) vs DB 행(ORM)** 을 구분해 설명한다.
6) **동시성·I/O**: 2·3단계가 왜 스레드 풀·동시성 상한을 쓰는지(문서 근거).
7) **변경 감지**: **목록 HTML·`news.txt` 각각에 대한 이중 HEAD**, **`ingest_state` 조회/갱신**, **`last_list_source`** — **플로우 상 어느 시점**에 DB를 읽고 쓰는지.
8) **운영·진단**: 환경변수 표(§8), **GET …/ingest-state**(상태 행 진단), Alembic 038(`ingest_state`)/039(`last_list_source`)를 **배포 체크리스트** 형태로 5줄 이내 언급.

금지: "똑똑한 비서" 같은 비유로 본문을 대체하지 말 것. DOM 표는 스펙에 따른 요약만.

출력: 소제목 8개에 대응하는 본문 + 마지막에 **한 단락 결론**(신규 온보딩이 읽어야 할 문서 순서: 스펙 §7 → §1.5 → §9).
```
