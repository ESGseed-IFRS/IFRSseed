# SR 인덱스 표 추출 — Docling 실패 시 대안 파서·서비스

Docling은 **표 구조·셀 단위**에 강하지만, 특정 PDF(인코딩, 폰트, 복잡 레이아웃)에서 실패할 수 있습니다.  
이때 **열·표를 구조적으로 복원**하는 데 자주 쓰이는 옵션을 정리합니다. (상용·오픈소스 혼합)

## 오픈소스 (로컬/온프레)

| 도구 | 특징 | 비고 |
|------|------|------|
| **pdfplumber** | 텍스트·좌표 기반, **표 후보 영역**·줄 단위 추출에 적합 | 룰/휴리스틱으로 다단 인덱스 커스텀 가능 |
| **pymupdf (fitz)** | 렌더링·블록·라인 단위, 속도 좋음 | 표 “의미”는 후처리 필요 |
| **Camelot** | **격자형 표(lattice)** / 스트림형 | 보더 있는 표에 유리, 레이아웃 까다로우면 실패 |
| **tabula-py** | Java Tabula 래퍼 | 표 추출 전통적 선택 |
| **Unstructured** | 파이프라인·partition PDF | 전처리·청크용, 표만 특화는 아님 |
| **Marker** / **nougat** 등 | MD/HTML 변환 | LlamaParse와 유사 축, 품질은 문서 의존 |

## 클라우드·API (레이아웃·표 모델)

| 서비스 | 특징 |
|--------|------|
| **Azure AI Document Intelligence** (구 Form Recognizer) | 표·키-값·레이아웃, 한국어 SR에 자주 사용 |
| **AWS Textract** | 표·폼 추출 |
| **Google Document AI** | 동일 계열 |

이들은 **셀/바운딩 박스**를 주기 때문에, GRI 다단 인덱스를 **열 단위**로 나누는 데 Docling보다 안정적인 경우가 많습니다.

## 현재 코드베이스와의 관계

- **LlamaParse**: 페이지 → 마크다운(또는 유사 텍스트). 레이아웃이 깨지면 **LLM 한 번**에 더 의존 → `sr_index_plain_text`의 줄 번호·**맥락 접두어**·**오른쪽 열 2차 패스**로 보완.
- **Docling**: 성공 시 표 구조 우선 병합(`multi_parser_merger`). 실패 시 위 보완 경로만 사용.

## 선택 가이드 (요약)

1. **인덱스만** 빠르게 안정화: **Azure Document Intelligence** 또는 **pdfplumber + 열 분할 휴리스틱** 프로토타입.
2. **비용 없이**: **pdfplumber** / **pymupdf**로 좌표 수집 후, GRI 인덱스 전용 **열 스플릿** 규칙을 코드화.
3. **이미 LlamaParse 사용 중**: 지금처럼 **LLM 2패스(오른쪽 열)** + 프롬프트 검수로 먼저 개선하고, 여전히 부족하면 API 파서를 **Docling 병렬**에 추가하는 형태가 단계적입니다.
